# 标签治理规则与映射模板

## 检查清单

对任务数据中存在的标签体系，检查以下一致性维度：

| 检查维度 | 问题信号 | 说明 |
|---------|----------|------|
| 命名一致性 | 大小写混用、含空格、特殊字符不统一 | 同一项目中应遵循同一套命名约定 |
| 分隔符一致 | 同时存在多种层级分隔方式 | 如 `/` 与 `:` 混用，或有的有层级、有的无层级 |
| 拼写正确性 | 明显拼写错误 | 如 `opearation` → `operation` |
| 维度互斥性 | 同一任务携带多个同维度标签 | 一个问题通常只应属于一个分类维度 |
| 粒度合理性 | 单维度下标签数量过度膨胀或过度集中 | 过多则分类失效，过少则失去区分度 |

> **原则**：不做"应该用哪种命名风格"的硬性规定，只检查"同一项目内是否自洽"。

## 常见维度参考

以下维度是多数团队可能用到的分类方向，**非强制标准**。团队应根据自身业务特点选择或自定义：

| 维度类型 | 典型用途 | 说明 |
|---------|----------|------|
| 技术域 / 业务域 | 标识任务所属模块或领域 | 如 `area/*`、`domain/*`、`module/*` |
| 关联项目 / 产品 | 标识跨项目归属 | 如 `project/*`、`product/*` |
| 环境 | 标识影响范围 | 如 `env/*`（prod/staging/dev） |
| 严重等级 | 标识优先级或紧急程度 | 如 `severity/*`、`priority/*` |
| 来源 | 标识任务如何产生 | 如 `source/*`（用户反馈/监控告警/技术债） |
| 工作项类型 | 补充任务分类 | 如 `type/*`（feature/bug/doc/refactor） |
| 流程状态 | 补充工作流阶段 | 如 `status/*`（设计/开发/验收） |

**关键判断**：
- 如果团队未使用标签，或标签使用率极低，**不应输出标签治理建议**
- 如果团队只有少量标签（如 `< 5 个`），重点关注命名一致性即可
- 如果某维度下标签数量极多，建议检查是否存在过度细分，或是否应合并/升维

## 标签膨胀处理参考

当发现某个维度下标签数量过多时，可按以下思路分析：

1. **识别可归并的子标签**：功能相近、边界模糊、 rarely used 的标签考虑合并
2. **检查是否有临时/一次性标签**：如带日期、带版本号的标签，通常应清理
3. **评估是否需要升维**：如 `area/*` 下某类标签数量显著多于其他，可能应独立为新维度

> **示例**（仅作思路参考，非标准方案）：
> 某团队 `area/*` 下原有 40+ 个标签，按技术域归并为 `compute`、`storage`、`network`、`security`、`observability`、`platform`、`delivery`、`operations` 等 8 个。实际归并方案需由团队根据业务架构自行决定。

## 分析输出格式

只读分析阶段的输出应包含：

1. **标签概况**：总标签数、有标签的任务占比、各维度分布
2. **一致性问题**：列举发现的命名、分隔符、拼写等问题
3. **维度健康度**：各维度下的标签数量，异常膨胀的标注
4. **建议动作**：待人工确认的标签归并/重命名/清理建议

> **注意**：task-patrol 为只读分析，不涉及任何标签的创建、修改或删除操作。如需执行治理动作，需由用户在项目管理平台中人工完成或使用具有写权限的工具。
