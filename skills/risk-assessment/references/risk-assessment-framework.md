# 风险评估框架与计算公式

## 交付预测模型

### 基础公式

```
历史平均完成率 = SUM(最近 N 个 sprint 的 completed) / SUM(最近 N 个 sprint 的 total)
历史平均速率   = SUM(最近 N 个 sprint 的 completed) / N
预测完成周数   = 当前迭代剩余任务数 / 历史平均速率
风险颜色       = 预测完成周数 > 剩余周数 ? 红色 : (预测完成周数 > 剩余周数 * 0.8 ? 黄色 : 绿色)
```

### 数据来源

- `completed` / `total`：来自 `get_sprint_velocity` 返回的每个 sprint 的 `stats.{category}.completed` / `.total`
- 当前迭代剩余任务数：来自 `get_sprint_overview` 或 `search_workitems`（过滤当前 sprint + 非 DONE 状态）
- 剩余周数：版本截止日期 - 当前日期

### 趋势判断

| 信号 | 判断方法 | 含义 |
|------|---------|------|
| 完成率持续下降 | 最近 3 个 sprint 完成率逐个降低 | 团队速率衰减，可能有外部干扰 |
| 完成率波动大 | 标准差 > 平均值的 30% | 估算不准或需求变更频繁 |
| 完成率稳定 | 最近 3 个 sprint 完成率差异 < 10% | 团队节奏可预测 |

## 阻塞风险模型

### 关键路径识别

```
关键阻塞者 = 阻塞最多下游任务的前 N 个任务
阻塞深度   = 从该任务到最下游叶子节点的最长依赖链
```

### 数据来源

- `blocked.workitems`：被阻塞的任务列表（来自 `get_blocker_analysis`）
- `blocking.workitems`：阻塞他人的任务列表（来自 `get_blocker_analysis`）

### 阻塞风险等级

| 等级 | 条件 | 建议 |
|------|------|------|
| 红色 | 关键阻塞者 > 0 且阻塞深度 > 2 | 立即升级，协调资源 |
| 黄色 | 被阻塞任务数 > 总任务数 10% | 跟踪进度，准备绕过 |
| 绿色 | 被阻塞任务数 = 0 | 正常推进 |

---

## 容量风险模型

### 过载判断

```
人均任务数 = 总活跃任务数 / 活跃成员数
过载阈值   = 人均任务数 * 1.5
过载成员   = 任务数 > 过载阈值的成员
```

### 数据来源

- `totalAssigned`：每个成员的分配任务数（来自 `get_member_workload_trend`）
- `overdueCount`：每个成员的逾期任务数（来自 `get_member_workload_trend`）

### 容量风险等级

| 等级 | 条件 | 建议 |
|------|------|------|
| 红色 | 过载成员数 > 总成员数 30% | 重新分配或延期 |
| 黄色 | 过载成员数 > 0 但 <= 30% | 协助拆分或转移 |
| 绿色 | 过载成员数 = 0 | 维持当前分配 |

---

## 综合风险评分

```
综合风险 = MAX(交付风险, 阻塞风险, 容量风险)
```

即三个维度中最高的风险等级作为项目整体风险等级。

---

## 动态阈值原则

所有阈值必须基于实际历史数据计算，禁止写死：

| 指标 | 动态计算方法 | 禁止做法 |
|------|------------|---------|
| 团队速率 | 历史 sprint 平均完成数 | 假设每人每天完成 X 个任务 |
| 容量上限 | 历史最大负载 × 1.2 | 设定固定任务数上限 |
| 阻塞容忍 | 历史平均解决时间 × 2 | 设定固定天数阈值 |
| 逾期判断 | 基于 finishTime 与当前日期比较 | 假设所有任务周期相同 |
